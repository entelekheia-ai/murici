## Summary

Introduce `KnowledgeRecord` as the first-class persistent unit of memory in Murici. Every fenced code block produced by the assistant in a chat session is automatically extracted, stored in a new `knowledge` IndexedDB store, and asynchronously enriched with a title and semantic summary. This is the v0 foundation of the Multi-layered Artifact Graph (MAG) architecture.

---

## Motivation

Murici currently stores only conversations and messages. Every output generated by the LLM — code, analysis, documents, reports — is embedded in the message stream and lost when the conversation is archived or the user forgets where it was.

The end goal (see RFC-0012 in dot-agent-spec) is a cognitive graph where Conversations, Agents, and Knowledge nodes are first-class, navigable, and retrievable by the LLM via `remember()`. This RFC implements the lowest layer of that goal: persistent knowledge nodes extracted from chat output, linked to their origin conversation, and enriched with metadata for future graph traversal.

The guiding insight from the MAG research: **the Artifact (achieved output), not the conversation, is the unit of memory**. Sessions exist to build knowledge nodes; once built, the node outlives the session.

---

## Scope

**In scope (v0):**
- `KnowledgeRecord` TypeScript interface
- IndexedDB migration v1 → v2 (new `knowledge` store)
- Extraction of fenced code blocks from assistant messages post-save
- Auto-title and summary generation (background LLM call)
- React Context integration (`knowledge[]` + `setKnowledge`)
- `originConversationId` as immutable birthplace of the node

**Out of scope (v1+):**
- Graph edges between KnowledgeRecords (`derivedFrom` populated by kernel)
- `agentRuns` roles beyond `"produced"` (populated when RFC-0014 `complete` effect exists)
- `.remember()` LLM tool for graph traversal
- Semantic grouping / cluster detection
- Multi-conversation node refinement (edges added when a later session touches the node)
- `memory_summarizer.agent` background synthesis pipeline

## Data Model

### `GeneralContent`

The v0 payload type for all knowledge nodes extracted from fenced blocks. Future DSL-declared types (e.g. `BookingConfirmation`) will replace this payload when RFC-0014 `complete ... with Type` is implemented in the kernel.

```ts
interface GeneralContent {
  language: string  // raw fence tag: "typescript", "md", "json", "sql", ""
  content: string   // full text inside the fence block
}
```

`language` is stored as the raw fence tag (free string). Normalization to RFC-0005 format annotations (`text(markdown)`, `text(json)`) is deferred to when `GeneralContent` becomes a `std.GeneralContent` stdlib type.

### `AgentRunRef`

```ts
interface AgentRunRef {
  agentId: string  // full agent ID per agent-id spec: "ns/name:v~digest" or "unknown/name"
  runAt: string    // ISO 8601
  role: "produced" | "consumed" | "transformed"
}
```

`agentId` encodes name implicitly (parseable as `name` = segment after last `/`). No separate `agentName` field is stored. In v0, `role` will always be `"produced"` — `consumed` and `transformed` are populated when sub-agent orchestration exists.

### `KnowledgeRecord`

```ts
interface KnowledgeRecord {
  id: string                        // uuid v4
  nodeType: "knowledge" | "document" | "task" | "automation" | "entity"
  // nodeType is always "knowledge" in v0.
  // Other values are reserved for future node types per RFC-0012.

  originConversationId: string      // immutable: the conversation where this node was created
  messageId: string                 // assistant message that contained the fenced block
  sourcePromptMessageId: string | null  // the user message that preceded it (the intent)

  title: string                     // display label, human-editable
  summary: string | null            // one-sentence semantic digest for LLM traversal; null until enrichment completes

  outputType: string                // "GeneralContent" in v0; DSL type name when RFC-0014 exists
  payload: GeneralContent           // typed by outputType; replaced by typed DSL payload in v1+

  derivedFrom: string[]             // KnowledgeRecord ids this node was built from; [] in v0
  agentRuns: AgentRunRef[]          // agents that touched this node; [] in v0 plain chat

  createdAt: string                 // ISO 8601
}
```

**On `originConversationId` vs. multi-conversation refinement:** a single knowledge node can be refined across multiple conversations in v1+ (the N:M:N model from the MAG spec). The `originConversationId` is always the creation context and is immutable. Subsequent cross-conversation touches are modelled as graph edges outside this record, not as additional fields on the record itself.

## IndexedDB Migration

**Current version:** 1 (stores: `conversations`, `messages`, `customModels`, `settings`)
**New version:** 2 (adds: `knowledge`)

```ts
// lib/local-db/schema.ts — upgrade block
upgrade(db, oldVersion) {
  if (oldVersion < 2) {
    const knowledge = db.createObjectStore("knowledge", { keyPath: "id" })
    knowledge.createIndex("by_conversation", "originConversationId")
    knowledge.createIndex("by_created", "createdAt")
    knowledge.createIndex("by_type", "nodeType")
  }
}
```

No migration of existing data is needed — the store starts empty.

## Extraction Algorithm

Triggered after `saveMessage()` completes for an **assistant** message. Does not run on user or system messages — those are inputs, not outputs. Fenced blocks only appear in assistant responses (the LLM generates them); user messages are the human's questions and system messages are configuration injected by the app, neither of which contain generated artifacts.

```
extractKnowledge(message: MessageRecord, conversationId: string):
  1. Match all fenced blocks: /```(\w*)\n([\s\S]*?)```/g
  2. For each match where content.trim().split('\n').length >= 3:
     a. Create KnowledgeRecord with:
        - id: uuid()
        - nodeType: "knowledge"
        - originConversationId: conversationId
        - messageId: message.id
        - sourcePromptMessageId: previousUserMessage.id | null
        - title: deriveTitle(language, content)  // see Title Derivation
        - summary: null  // populated by background enrichment
        - outputType: "GeneralContent"
        - payload: { language, content }
        - derivedFrom: []
        - agentRuns: []  // populated by SDK if agent was running
        - createdAt: now()
     b. Save to IndexedDB `knowledge` store
     c. Append to React Context `knowledge[]` via setKnowledge
  3. If any records were created, trigger background enrichment
```

**Minimum content threshold:** 3 non-empty lines. Prevents saving trivial one-liners (e.g. `` ```bash\nls\n``` ``).

## Title Derivation

Applied synchronously at extraction time to produce an immediate placeholder title.

| Content type | Rule |
|---|---|
| `language = "md"` or `language = "markdown"` | First `#` heading line, stripped of `#` prefix. Fallback: first non-empty line. |
| `language = "html"` | Content of `<title>` tag if present; else first `<h1>` text content. |
| All other languages | `"{language} · {HH:mm}"` placeholder — replaced by background enrichment. |
| Empty language tag | `"text · {HH:mm}"` placeholder. |

---

## Background Enrichment

After extraction, a background call generates `title` (for non-markdown) and `summary` for all nodes.

### Model selection

| Active model type | Behavior |
|---|---|
| Local (auto-discovered) | Fire immediately using the active model |
| Cloud (any hosted provider) | Do not fire automatically — show "✦ name" button on the node in UI |

The button triggers the same enrichment call on demand. Once triggered, the result is saved and the button disappears.

### Prompt

```
Give a short filename (no extension, max 5 words) and a one-sentence description for this content.
Respond as JSON: { "title": "...", "summary": "..." }

Content:
{payload.content}
```

### On response

```ts
await db.put("knowledge", { ...record, title: result.title, summary: result.summary })
setKnowledge(prev => prev.map(k => k.id === record.id ? { ...k, ...result } : k))
```

The UI reflects the update reactively via Context.

## Relationship to MAG Architecture

This RFC implements **Layer 1 — Nodes** of the MAG abstraction hierarchy defined in RFC-0012:

> *Individual memories and entities. The user sees each node explicitly.*

The `derivedFrom` field enables **Artifact Lineage** (Section 4.2 of the MAG spec) once the kernel emits `complete` effects per RFC-0014. The `agentRuns` field enables the **Agent → Artifact** authored-by edge.

The `summary` field is the semantic handle used by the future `.remember()` tool (Section 4.3 of the MAG spec): the Runtime searches `summary` text across nodes (hybrid: filter by `nodeType`/`outputType`, then semantic match) before returning lineage context to the LLM.

## Implementation Notes

These decisions were made during v0 implementation and diverge from or refine the original spec above.

### Title derivation and LLM enrichment are independent

The original spec implied LLM enrichment replaced placeholder titles. The implementation separates them:

- **Title** is always derived deterministically at extraction time (rules above). Placeholder titles (`"typescript · HH:mm"`) are replaced by the LLM. Derived titles (markdown `#` heading, HTML `<title>`) are preserved.
- **Summary** is always generated by the LLM for local models, regardless of how the title was obtained. A deterministic title does not skip LLM enrichment.

This is implemented in `lib/knowledge/enrich.ts` via `PLACEHOLDER_TITLE_RE` — the LLM result only overrides `title` when the current title matches the placeholder pattern.

### Local model detection

`isLocalModel` checks `modelData.provider === "local"` (auto-discovered models) OR `!!modelData.baseUrl` (manually configured custom models). This matches the fallback logic in `handleLocalChat` — models that rely on `NEXT_PUBLIC_OLLAMA_URL` without an explicit `baseUrl` in the LLM object are still correctly identified as local.

## Open Questions

1. **Extraction during agent runs:** when the SDK kernel is active, should `agentRuns` be populated at extraction time by reading the active `AgentSession`? Or deferred to when the kernel emits an explicit effect? *Tentative: read active session at extraction time for v0 — no kernel changes needed.*

2. **Title editability UX:** where exactly is the edit affordance exposed — inline in the knowledge node chip in the right panel, or only in the graph/list view? *Deferred to RFC-0003.*

3. **Minimum content length:** 3 lines is a heuristic. Should there be a character floor too (e.g. 50 chars)? *Open.*
