Relatório técnico · 2026-07-11

Ollama → murici

O que o servidor Ollama expõe hoje — nativo e OpenAI-compat — que o murici ainda não consome, e por que boa parte dessa lacuna não precisa de PR upstream: é arquitetural do lado do cliente.

fonte: github.com/ollama/ollama · MIT · público consumidor: github.com/entelekheia-ai/murici
Resumo

O essencial

6campos de timing na API nativa (Metrics), todos em nanossegundos
3campos sobrevivem na camada OpenAI-compat que o murici usa hoje
/api/psendpoint de "modelos ativos" — público, sem auth de admin
0rota de PR necessária pra maioria dos ganhos — é troca de endpoint, não de servidor

Diferente do relatório anterior sobre oMLX, aqui a notícia é melhor: a telemetria rica do Ollama — load_duration, prompt_eval_duration, eval_duration, modelos ativos com VRAM/RAM, progresso de download — é toda pública, sem cookie de admin, sem chave especial. O problema não é acesso, é rota: o murici já distingue Ollama na descoberta (probeOllama, endpoint nativo /api/tags), mas depois joga tudo pro mesmo caminho genérico OpenAI-compat, que descarta as durações e devolve só prompt_tokens/completion_tokens/total_tokens.

Isso muda o formato das recomendações: a maior parte é "trocar de endpoint no cliente", não "esperar um PR ser aceito". As duas exceções reais — tokens/s pré-computado e visibilidade de cache hit — viram propostas de PR na seção 4.

01 · Superfície do Ollama

Nativo vs. OpenAI-compat: dois níveis de detalhe

Go, sem auth por padrão em instalação local. Duas famílias de rota coexistem no mesmo processo: a API nativa do Ollama (/api/*) e uma camada de compatibilidade OpenAI/Anthropic (/v1/*) — a segunda é um adaptador fino sobre a primeira, e descarta a maior parte dos campos de timing ao converter.

EndpointCamadaDado relevante para murici
POST /api/chatnativaMetrics completo: total_duration, load_duration, prompt_eval_count, prompt_eval_duration, eval_count, eval_duration — tudo em nanossegundos
POST /api/generatenativaMesmo Metrics, mais context[] (deprecated) pra continuação sem re-enviar histórico
GET /api/psnativaModelos carregados agora: size, size_vram, context_length, expires_at (keep_alive) — sem auth de admin
POST /api/shownativaDetails: family, parameter_size, quantization_level, context_length; mais ModelInfo com metadados GGUF crus
GET /api/tagsnativaLista de modelos instalados — já é o que probeOllama usa hoje na descoberta
POST /api/pull · /api/pushnativaNDJSON de progresso: status, digest, total, completed (bytes)
POST /v1/chat/completionsOpenAI-compatÉ o que o murici usa hoje. usage só com prompt_tokens/completion_tokens/total_tokens — todas as durações são descartadas na conversão
POST /v1/messagesAnthropic-compatMesma limitação; cache_control de prompt caching explicitamente não suportado

O formato nativo — e a armadilha das unidades

Toda resposta de /api/chat e /api/generate embute isto (nanossegundos, não milissegundos — docs/api.md:27-29 avisa explicitamente):

{
  "total_duration": 10706818083,
  "load_duration": 12871000,
  "prompt_eval_count": 26,
  "prompt_eval_duration": 267510000,
  "eval_count": 298,
  "eval_duration": 10396087000,
  "done": true,
  "done_reason": "stop"
}

Tokens/s não vem pronto — o próprio doc do Ollama documenta a fórmula manual: tokens/s = eval_count / (eval_duration / 1e9). Isso é o inverso do oMLX, que já entrega generation_tokens_per_second calculado.

Cache: existe internamente, mas some antes de chegar na API

O processo llama-server por trás do Ollama já sabe quanto do prompt veio do cache — internamente ele reporta cache_n (tokens do cache) e prompt_n (tokens novos) separados. Mas o handler do Ollama soma os dois (promptEvalCount() = CacheN + PromptN, llm/llama_server.go:1456-1462) antes de expor como prompt_eval_count — o split se perde. Não existe, em nenhuma rota, um equivalente ao cached_tokens do oMLX/OpenAI.

Hardware: computado, nunca servido

O pacote discover/ já sonda GPU/VRAM por dispositivo pra decidir onde alocar cada modelo (server/sched.go) — mas nenhuma rota HTTP expõe isso. /api/ps só devolve os agregados size (total) e size_vram (quanto está na GPU); a diferença dá uma aproximação de RAM, mas não há split por GPU nem inventário de hardware via API.

02 · Estado atual

O que o murici já faz com Ollama — mais do que com oMLX

  • Ollama já é tratado como caso especial na descoberta — diferente do oMLX. app/api/models/discover/route.ts:70-95 tem um probeOllama() dedicado que chama /api/tags nativo, não o genérico /v1/models.
  • NEXT_PUBLIC_OLLAMA_URL (default http://localhost:11434) é usado como fallback em três lugares: discover/route.ts:104-105, components/chat/chat-helpers/index.ts:46,590,650, e lib/runtime/headless-runner.ts:27.
  • Mas isso para na descoberta. Uma vez que o modelo é listado, ele entra no mesmo balde genérico provider: "local" que oMLX, LM Studio e Oobabooga — não existe valor "ollama" em nenhum enum (types/llms.ts, db/models.ts, lib/local-db/models.ts: zero hits).
  • O chat de fato roda por app/api/chat/custom/route.ts, com createOpenAI({baseURL, apiKey}) do Vercel AI SDK — sempre a camada compat, nunca /api/chat nativo. É por isso que os campos de duração nunca chegam: eles são descartados no openai/openai.go do próprio Ollama antes mesmo da resposta sair do servidor.
  • Mesma lacuna de UI do relatório de oMLX: nenhuma ocorrência de usage, tokensPerSecond ou duration em components/ — o problema de exibição é idêntico nos dois casos, só a fonte de dado muda.
Ponto de atenção: como não existe um marcador engine: "ollama" persistido no registro do modelo, o murici hoje não tem como saber, no momento do chat, se deve chamar /api/chat nativo (Ollama) ou só /v1/chat/completions (oMLX e outros). Chamar a rota nativa errada em outro servidor local dá 404. Isso é pré-requisito de qualquer recomendação abaixo — ver plano na seção 5.

04 · Contribuição upstream

Vale a pena um fork ou PR pro Ollama?

O Ollama já é o projeto oficial (ollama/ollama, MIT, altíssima atividade — centenas de branches de contribuidores). Como no caso do oMLX, não há "abrir" a fazer — só propor PRs escopados pras duas lacunas reais que não dá pra contornar no cliente:

1. Pré-computar tokens_per_second e adicionar ao Metrics

Hoje todo cliente reimplementa a mesma divisão (eval_count / (eval_duration/1e9)) documentada em texto no docs/api.md. Adicionar eval_tokens_per_second e prompt_tokens_per_second ao struct Metrics é aritmética pura sobre campos que já existem — baixíssimo risco. O oMLX já faz exatamente isso; citar como prior art ajuda a justificar o PR.

2. Separar cache_n de prompt_eval_count na resposta pública

O dado já existe em llamaServerTimings.CacheN — é só parar de somar antes de serializar. Expor como campo novo (cached_tokens, seguindo a nomenclatura que OpenAI e oMLX já usam) sem remover prompt_eval_count mantém compatibilidade retroativa. Maior valor do PR: nenhum cliente Ollama hoje consegue mostrar "esse prompt reaproveitou cache", inclusive concorrentes do murici.

3. Expor breakdown de GPU em /api/ps

runnerRef.gpus []ml.DeviceID já existe internamente em server/sched.go. Adicionar um array opcional gpus: [{id, size_vram}] ao ProcessModelResponse é estender um struct já serializado, sem tocar em lógica de agendamento.

Diferença importante em relação ao relatório de oMLX: nenhuma dessas propostas desbloqueia dado hoje inacessível por auth — são melhorias de ergonomia (poupar o cliente de recalcular ou de perder informação já computada). Prioridade sugerida: item 2 primeiro (é o único gap de informação, os outros dois são conveniência).

05 · Plano de implementação

Por onde começar no murici

  1. Marcar o engine na descoberta. Em app/api/models/discover/route.ts, os modelos vindos de probeOllama() (linhas 70-95) devem carregar um marcador engine: "ollama" (hoje só existe provider: "local") — sem isso, o roteamento de chat não tem como saber se pode chamar /api/chat nativo com segurança.
  2. Nova rota de chat nativa, paralela a app/api/chat/custom/route.ts (ex.: app/api/chat/ollama/route.ts), que chama {base_url}/api/chat diretamente em vez de passar pelo client OpenAI-compat do AI SDK — só assim os campos de Metrics sobrevivem. Alternativa com menos código: avaliar o provider comunitário ollama-ai-provider pro Vercel AI SDK, que mapeia os campos nativos pra providerMetadata; checar compatibilidade com AI SDK v5 antes de adotar.
  3. Propagar como data-ollama-metrics, reaproveitando o mesmo padrão de UI-message-part já usado por data-debug em lib/server/agent-stream.ts — mesmo mecanismo recomendado no relatório de oMLX, mesma renderização em components/messages/message.tsx.
  4. Hook de polling para GET /api/ps, reaproveitável no mesmo painel de "atividade" sugerido pro /api/status do oMLX — os dois podem compartilhar componente de UI, com adaptadores diferentes por engine.
  5. Mesma RFC de destino. Como no caso oMLX, isso encaixa como continuação de project/rfc/0001-local-model-autodiscovery.md, que já teve o campo "VRAM / hardware metrics" deferido — uma RFC única cobrindo os dois engines evita duas propostas de design concorrentes pra UI que vai ser, na prática, a mesma.
