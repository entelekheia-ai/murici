Relatório técnico · 2026-07-11

oMLX → murici

O que o servidor oMLX expõe hoje (API pública e painel admin) que o murici ainda não consome — tokens/s, cache, memória — e onde vale a pena abrir PR upstream em vez de contornar no cliente.

fonte: github.com/jundot/omlx · Apache-2.0 · público consumidor: github.com/entelekheia-ai/murici
Resumo

O essencial

9campos de timing/tokens já na resposta OpenAI-compat (Usage)
0desses campos hoje lidos ou exibidos no murici
/api/statusendpoint público ideal para polling leve
admin-onlytelemetria rica (cache hot/cold, tok/s ao vivo) fica atrás de cookie de sessão

O oMLX já calcula e devolve, em toda resposta de /v1/chat/completions e /v1/completions, um bloco Usage estendido com time_to_first_token, prompt_tokens_per_second, generation_tokens_per_second e prompt_tokens_details.cached_tokens — dado pronto, autenticado só com API key, que o murici simplesmente descarta hoje (lib/server/agent-stream.ts nunca lê result.usage). Esse é o menor esforço com maior retorno.

A telemetria mais rica que aparece nos screenshots do README do oMLX (dashboard de modelos ativos, cache hot/cold, benchmark) vive em /admin/api/*, que exige cookie de sessão de admin — não dá pra alcançar só com API key. Isso define duas trilhas: implementar agora (dados já públicos) e propor upstream (dados que existem no servidor mas não têm rota pública).

01 · Superfície do oMLX

O que já está acessível via API key

FastAPI, dois domínios de auth: /v1/* e /api/status aceitam Bearer/API-key; /admin/* exige cookie de sessão de browser. Sub-keys criadas via /admin/api/sub-keys continuam presas ao domínio API-key — não abrem /admin/api/*.

O campo que mais importa: Usage em /v1/chat/completions

Extensão própria do oMLX sobre o schema OpenAI (api/openai_models.py:345-360), presente em toda resposta — streaming (chunk final, requer stream_options.include_usage) ou não:

{
  "prompt_tokens": 812, "completion_tokens": 214, "total_tokens": 1026,
  "prompt_tokens_details": { "cached_tokens": 640, "audio_tokens": 0 },
  "model_load_duration": 0.0,
  "time_to_first_token": 0.31,
  "total_time": 4.82,
  "prompt_eval_duration": 0.31,
  "generation_duration": 4.51,
  "prompt_tokens_per_second": 2619.4,
  "generation_tokens_per_second": 47.4
}

Ressalva: essa extensão só existe no caminho OpenAI-compatível. /v1/messages (Anthropic-compat) devolve um AnthropicUsage mais pobre — input_tokens, output_tokens, cache_creation_input_tokens, cache_read_input_tokens — sem TTFT nem tok/s.

Cache: o que dá pra ver, o que não dá

O oMLX implementa um KV cache em camadas (RAM "quente" + SSD "fria", com prefix sharing) — é o que aparece em omlx_hot_cold_cache.png. As estatísticas detalhadas (hit_rate, tokens_saved, hot_cache_entries, save_rate) vivem em cache/stats.py mas só chegam a /admin/api/stats. Via API key, o único sinal de cache é total_cached_tokens/cache_efficiency em /api/status e prompt_tokens_details.cached_tokens por resposta — dá pra construir um indicador "X% do prompt veio do cache" por mensagem, mas não o breakdown hot/cold completo do dashboard.

Dado computado mas descartado no próprio oMLX

Vale registrar como achado, não só como gap do murici: o decoding especulativo DFlash calcula acceptance_ratio e cycles_completed por request (engine/dflash.py:1004-1042) e só os manda pro log — nunca chegam a Usage nem a nenhum endpoint admin. O próprio docs/experimental/dflash_mlx_integration.md:294-295 lista isso como trabalho futuro não feito.

02 · Estado atual

O que o murici faz hoje

  • Conexão com oMLX já existe, mas genérica: app/api/models/discover/route.ts sonda ~/.omlx/settings.json e GET {base_url}/v1/models, mapeando pra provider: "local" — não existe um provider "omlx" dedicado (decisão deliberada da RFC-0001, porque vLLM disputa a mesma porta 8000).
  • O request de chat (app/api/chat/custom/route.ts) usa o client OpenAI-compat do Vercel AI SDK apontado pro base_url local, forçando .chat() em vez da Responses API.
  • result.usage do streamText() nunca é lido em lib/server/agent-stream.ts:77-105 — o stream vai direto pra toUIMessageStream() sem passar os campos de uso adiante.
  • Zero ocorrências de usage, promptTokens, tokensPerSecond ou TTFT em components/, lib/ ou app/ — confirmado por grep. Nenhuma UI de métricas existe hoje, nem embrionária.
  • Existe um precedente estrutural pronto pra reusar: agent-stream.ts já emite uma UI-message-part customizada, data-debug, carregando metadados fora do fluxo de texto. O mesmo padrão serve pra uma data-omlx-metrics.
  • RFC-0001 (project/rfc/0001-local-model-autodiscovery.md), que documenta a auto-descoberta de modelos locais, já lista "VRAM / hardware metrics" explicitamente como fora de escopo da Fase 1 — ou seja, isso é uma continuação natural planejada, não uma ideia nova.

04 · Contribuição upstream

Vale a pena um fork ou PR pro oMLX?

O oMLX já é público (jundot/omlx, Apache-2.0, ativo — commits e PRs numerados recentes, docs/CONTRIBUTING.md com fluxo de fork padrão). Não faz sentido "abrir" o projeto — ele já está aberto. O que faz sentido é propor PRs pontuais e escopados que resolvem exatamente a lacuna acima: dado que já existe no servidor, mas não tem rota pública. São mudanças pequenas e de baixo risco porque não tocam no motor de inferência, só na camada de API.

1. Escopo de API key pra métricas somente-leitura

Hoje /admin/api/sub-keys já cria chaves com permissão restrita à API pública. Adicionar um escopo novo (ex.: metrics:read) que libera uma versão somente-leitura de /admin/api/stats — sem os endpoints de mutação (settings, cache clear, bench) — resolveria o maior gap sem expor superfície administrativa. Por quê topam: é aditivo, não muda comportamento default, e o próprio produto se beneficia de mais integrações third-party puxando telemetria sem precisar da senha de admin.

2. Paridade de Usage entre /v1/messages e /v1/chat/completions

A extensão de timing (time_to_first_token, *_tokens_per_second) só existe no schema OpenAI. Estender AnthropicUsage com os mesmos campos é praticamente copiar código já existente pra outro serializer — baixíssimo risco, alta chance de aceite.

3. Expor acceptance_ratio do DFlash

O próprio doc do projeto (dflash_mlx_integration.md:294-295) já lista isso como trabalho futuro desejado — não é uma ideia externa, é preencher uma lacuna que os mantenedores já registraram. Adicionar ao Usage quando dflash_enabled está ativo.

4. Bits de quantização do modelo carregado em /v1/models/status

ModelInfo hoje só tem id, object, created, owned_by, max_model_len. A informação de quantização (oQ) só aparece em endpoints de job de (re)quantização admin — nunca pro modelo já servido. Um campo quant_bits/quant_method em /v1/models/status é pequeno e evita que clientes tenham que adivinhar pelo nome do arquivo.

Recomendação: começar pelo item 1 como PR isolado (é o que desbloqueia o dashboard de atividade ao vivo no murici) e usar os itens 2-4 como PRs de acompanhamento — cada um é pequeno o bastante pra revisar e mergear independente. Não é necessário fork de longo prazo; o fluxo de CONTRIBUTING.md (fork → branch → PR) é suficiente, dado que nenhuma dessas mudanças precisa divergir da branch principal.

05 · Plano de implementação

Por onde começar no murici

  1. Ler result.usage em lib/server/agent-stream.ts (perto da linha 77-105, ao lado do data-debug existente) e propagar como nova UI-message-part.
  2. Consumir a part nova em onData de components/utility/chat-handler-provider.tsx (linha ~219) e guardar no estado da mensagem.
  3. Renderizar em components/messages/message.tsx, na área de cabeçalho da mensagem (~linha 254-270) — hoje só tem ícone + nome, sem linha de metadados.
  4. Hook de polling novo para GET /api/status, consumido por um indicador leve no seletor de modelo ou sidebar.
  5. Documentar como RFC: já existe o gancho certo — project/rfc/0001-local-model-autodiscovery.md deferiu "VRAM / hardware metrics" pra depois. Uma RFC-000X nova, sucedendo esse item, evita retrabalho de design e mantém o histórico de decisão consistente com o resto do projeto.
